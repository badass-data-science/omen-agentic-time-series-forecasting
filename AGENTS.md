# Omen

Five layers of agentic time series tooling. Each layer is a FastMCP server
(typed tools) plus a companion OpenClaw skill (markdown playbook telling an
agent how to sequence those tools and what to report). Built to be driven
by OpenClaw + an LLM agent (developed against GLM-5.2 on Ollama Cloud, but
nothing is model-specific).

- `analyst`    -> `ts-analyst`    : explore a series, recommend an approach. No fitting.
- `forecaster` -> `ts-forecaster` : fit + backtest candidate models on a held-out window.
- `deploy`     -> `ts-deploy`     : retrain the chosen model on full data, forecast forward.
- `monitor`    -> `ts-monitor`    : compare deployed forecast vs. reality, detect drift, recommend retraining.
- `retrain`    -> `ts-retrain`    : on retrain_now, re-run analyst+forecaster and decide deterministically whether the new candidate beats what's deployed; redeploys only through the gated `execute_redeploy(confirmed=True)` tool, either human-approved (default) or via an explicitly authorized autonomous mode.

Full architecture/setup is in README.md â€” don't duplicate it here.

**Project naming**: the project/package/repo directory is `omen` (formerly
`agentic-ts-toolkit` â€” renamed, not rebranded from scratch). The five
layers deliberately KEPT their existing `ts-analyst`/`ts-forecaster`/
`ts-deploy`/`ts-monitor`/`ts-retrain` server names, skill names, and
MCP-qualified tool names (e.g. `ts-analyst__basic_stats`) unchanged â€”
that's an already-established, separately-branded layer referenced
everywhere (every `SKILL.md`, every test, every docstring, every blog
post), and renaming it alongside the project name would have been a much
larger, more disruptive change than what was asked for. If you're adding
new tools, keep using the `ts-*` naming convention for them; don't start
mixing in `omen-*`-prefixed names for consistency with the project name,
since that would just create two competing conventions.

## Commands

```bash
pip install -e ".[all,dev]"    # everything, including test deps
pytest                          # run the suite
ts-analyst-server                # run a layer's MCP server standalone (Ctrl+C to stop)
python -m build                 # build sdist/wheel for publishing
```

Per-layer installs: `.[analyst]`, `.[forecaster]`, `.[deploy]`, `.[monitor]`, `.[retrain]` if you don't want every dependency.

## Non-obvious conventions

- **One shared `data_prep.py`** at the package root (`omen.data_prep`). Every subpackage imports it â€” never re-add a per-layer copy.
- **Within a subpackage, tools return plain functions taking a DataFrame; `server.py` wraps them as `@mcp.tool()` and does the CSV loading.** Keep that split â€” it's what makes the tool logic unit-testable without spinning up MCP (see `tests/`, which import the tools modules directly, not through `server.py`).
- **`recommend_retraining` in `monitor_tools.py` is deliberately deterministic/rule-based**, not an LLM judgment call â€” that's intentional, not a missed opportunity to "let the agent decide." Keep new decision-style tools (accept/reject, threshold-based verdicts) rule-based for the same reason: reproducibility given the same inputs. `retrain_tools.py`'s `compare_candidate_to_deployed` follows the same rule.
- **Every `SKILL.md` references its tools by their MCP-qualified name** (`ts-analyst__basic_stats`, etc.), not by file path â€” these are OpenClaw skills, not exec-based scripts. If you add a tool, update the matching `SKILL.md`'s tool list too.
- **Tests use `pytest.importorskip("statsmodels")` / `"sklearn"`** in the layers that need them (`analyst`, `forecaster`, `deploy`), so `pip install -e .` (core deps only) doesn't break the whole suite. `monitor` and `data_prep` tests have no such guard â€” keep it that way, they're meant to run everywhere. `retrain` is mostly in that camp too (`record_deployment`/`load_deployment_manifest`/`compare_candidate_to_deployed` need nothing beyond core), but `test_retrain_tools.py`'s one test of `execute_redeploy`'s real dispatch path uses a per-test (not module-level) `importorskip`, since `execute_redeploy` alone needs the `deploy` extra â€” don't move that guard to the top of the file, it would skip tests that don't need it.
- **`ts-retrain` is the one place in the toolkit with durable state** â€” `record_deployment`/`load_deployment_manifest` read and write a JSON manifest (`deployment_manifest.json` by default, next to the series CSV; see `_manifest_path_for` in `retrain_tools.py`). Every other tool in every other layer is a pure function of its explicit inputs â€” don't add more file-based state elsewhere without a similarly strong reason.
- **`ts-retrain` only ever changes a deployment through `execute_redeploy`, which refuses to run without `confirmed=True`** (no default performs the action). Two paths reach that confirmation, both documented in `SKILL.md`: human approval in-conversation (default), or an explicitly authorized autonomous mode where the skill calls `execute_redeploy(confirmed=True)` itself. The authorization check for autonomous mode is a prose contract enforced by the skill's instructions, not something the code validates â€” `execute_redeploy` itself doesn't know or care which path led to `confirmed=True`. Don't weaken the skill's "default to human mode when ambiguous" language, and don't add a code path that sets `confirmed=True` by default anywhere.
  - Human-confirmed example: after `ts-retrain` reports `should_redeploy: true` and stops, the user says *"Go ahead and redeploy the candidate you just recommended"* â€” that follow-up is what authorizes the `execute_redeploy(..., confirmed=True)` call.
  - Autonomous-mode example: authorization is scoped per series, stated explicitly, e.g. *"For the `daily_demand.csv` series specifically, you're authorized to redeploy automatically whenever ts-retrain finds a candidate that beats the current deployment â€” no need to ask me first. Everything else still needs my confirmation as usual."* Absent wording this explicit and this scoped, treat autonomous mode as not authorized.
- **Both `SARIMAX(...)` call sites (`forecaster/model_tools.py::fit_sarima`, `deploy/forecast_tools.py::forecast_sarima`) pass a single-column DataFrame (`df[["value"]]`), not a Series (`df["value"]`).** This isn't stylistic -- statsmodels does an in-place `ndarray.shape=` reshape internally, but only for 1-D endog, and NumPy 2.5+ deprecated in-place shape mutation, so a Series triggers a `DeprecationWarning` on every fit. A single-column DataFrame is already 2-D, so statsmodels skips that code path -- verified identical AIC/BIC, residuals, and forecast values/shapes either way. If you add another `SARIMAX(...)` call site, pass a DataFrame the same way.

## Known modeling caveats (intentional, not bugs)

- `forecaster/model_tools.py`'s gradient-boosted-trees backtest is **one-step-ahead** (true lagged values); `deploy/forecast_tools.py`'s gradient-boosted-trees forecast is **recursive** (predictions feed back in as lags). These are not directly comparable to each other or to ETS/SARIMA's genuine multi-step forecasts â€” both modules' docstrings and return values flag this explicitly. Don't "fix" this asymmetry without understanding why it exists.
- `monitor/monitor_tools.py`'s `detect_data_drift` cannot distinguish an ongoing trend/seasonal transition from a genuine regime change â€” confirmed empirically that it flags `drift_detected=True` on trending synthetic data with zero injected anomaly. This is by design (a coarse, fast check); read `interpretation`, don't treat the boolean as ground truth.
  - It now reports `mean_shift_cohens_d` (pooled-SD Cohen's d for the Welch's-t mean-shift test â€” pooled SD is the standard choice for the effect size even though the test itself doesn't assume equal variances) plus the raw `ttest_statistic`/`ks_statistic`, which is a REAL BUG FIX, not just an enrichment: both statistics were already being computed (`ttest_stat, ttest_pvalue = ttest_ind(...)`, `ks_stat, ks_pvalue = ks_2samp(...)`) and then silently discarded before the function returned â€” only the p-values made it into the result. Confirmed on the project's own synthetic data that the effect size is what actually distinguishes the two "drift" cases the trend-only-false-positive caveat above conflates at the boolean level: the unmodified trending series alone comes back `mean_shift_cohens_dâ‰ˆ-0.42` (small-to-medium), while injecting an obvious +200 level shift on top of it pushes that to `â‰ˆ7.06` (enormous) â€” both flag `drift_detected=True`, only the effect size tells you which is which. `ks_statistic` needed no separate effect-size derivation (unlike KPSS's coarse p-value elsewhere in this project) since the KS statistic itself already IS a natural 0-to-1 magnitude (max distance between the two windows' empirical CDFs).
- `monitor/monitor_tools.py` gained `_bootstrap_metric_ci`/`compute_metrics_with_ci`, duplicated verbatim (same field names: `mae_ci_lower/upper`, `rmse_ci_lower/upper`, `mape_pct_ci_lower/upper`, `ci_confidence_level`, `ci_n_bootstrap`) from `forecaster/model_tools.py`'s versions rather than imported cross-layer â€” same precedent as `deploy/forecast_tools.py`'s `_aicc`/`_build_lag_features` duplication. `compare_forecast_to_actuals` now calls this instead of the bare `compute_metrics`; this is additive, not a rename, matching the same pattern documented for `forecaster/model_tools.py`'s `fit_*` CI additions â€” every pre-existing field name (`mae`, `rmse`, `mape_pct`) is unchanged, only new keys were added. `compute_metrics` itself (the plain, CI-less version) is UNCHANGED and still exported/tested directly â€” don't remove it or fold its logic into `compute_metrics_with_ci` only, other code may still want the cheap point-estimate-only path.
- `monitor/monitor_tools.py`'s `recommend_retraining` gained optional `mape_now_ci_lower`/`mape_now_ci_upper` params (fed from `compare_forecast_to_actuals`'s new `mape_pct_ci_lower/upper`). When both are supplied, it computes `pct_degradation_ci_lower/upper` by applying the SAME linear transform used for the point estimate (`100 * (mape_now - mape_backtest) / mape_backtest`) to each CI bound â€” valid because that transform is monotonic increasing in `mape_now` for a fixed positive `mape_backtest`, so the bounds map through directly without needing their own resampling. `mape_backtest` itself is treated as a FIXED constant throughout â€” no CI is combined in for it, even if the caller has one, since Layer 2's own backtest CI isn't threaded through this far and combining an uncertain `mape_backtest` would require its own variance-combination logic (see `deploy/forecast_tools.py::forecast_ensemble`'s equivalent problem) that wasn't judged worth the complexity here. `degradation_threshold_within_ci` flags when `error_degradation_threshold_pct` falls inside `[pct_degradation_ci_lower, pct_degradation_ci_upper]` â€” when true, a sentence is appended to `reasoning` (not a separate field to remember to check) saying the degraded/not-degraded verdict is sensitive to sampling noise in `mape_now`. All three new fields are `None` (not `0` or `False`) when the CI bounds aren't supplied â€” same "missing means None, not a fabricated default" convention used everywhere else in this project (e.g. `_check_forecast_plausibility`'s insufficient-history fields).
- `monitor/monitor_tools.py` gained `_wilson_score_interval` (binomial Wilson score CI, `scipy.stats.norm.ppf`-based) applied to `compare_forecast_to_actuals`'s `interval_coverage.empirical_coverage_pct` â€” chosen over the normal approximation specifically because coverage here is often computed from a SMALL number of matched dates (early in a monitoring cycle) and/or a proportion near 0% or 100%, both regimes where the normal approximation is known to behave badly (can produce bounds outside [0,1]) and Wilson doesn't. **This is deliberately supplementary, not a replacement for `well_calibrated`'s existing threshold logic** â€” `well_calibrated` stays exactly `abs(coverage_pct - nominal_pct) <= 15`, unchanged, because that same 15-percentage-point threshold is INTENTIONALLY mirrored in `forecaster/model_tools.py::_interval_coverage` (see below/above) and changing the definition here without also changing there would break a deliberate cross-layer consistency this project has already documented. If you ever do want a CI-aware `well_calibrated` (e.g. "well-calibrated if nominal falls inside the Wilson CI"), that's a bigger design change spanning both `compare_forecast_to_actuals` and `_interval_coverage` together, not a one-line edit here.
- `monitor/monitor_tools.py` gained `_residual_outliers`, called from `compare_forecast_to_actuals` and returned as its new `residual_outliers` field. Reuses the exact modified-z-score formula (median + MAD, Iglewicz & Hoya 1993, `z_threshold` default 3.5) from `analyst/analysis_tools.py::detect_anomalies_robust_zscore`, but applied to a single FIXED set of residuals (`actual - forecast` across all matched points) rather than a rolling window â€” a rolling window doesn't make sense here since the elapsed comparison horizon is typically short (often under 30 points, sometimes under 10). **Known limitation, documented in the function's own docstring, not silently left as a surprise**: if half or more of the residuals are EXACTLY identical (most realistically all exactly 0, i.e. suspiciously perfect forecasts on most days), the MAD itself degenerates to 0, and the code's `if mad > 0` guard falls back to an all-zero modified-z-score array â€” flagging nothing, even if one remaining residual is enormous. This is the SAME self-dilution failure class that motivated building the robust detector for `ts-analyst` in the first place (see the ADF/anomaly detector caveats above), just triggered by a different, rarer precondition (near-identical residuals rather than a single spike inflating a rolling window's own std). Caught via `tests/test_monitor_tools.py`'s own test-writing process â€” an early draft of the test used EXACT forecast matches for the "good" days, which is exactly the degenerate condition, and the test failed until the "good" days were given small non-zero misses instead. `metrics_excluding_outliers` is `None` (not an empty/zero metrics dict) whenever fewer than 2 non-outlier points remain or nothing was flagged as non-outlier â€” a real "can't compute this" state, not a fabricated comparison.
- `monitor/monitor_tools.py`'s `rolling_drift_check` reuses `detect_data_drift` directly (calling it once per check on a truncated prefix of `df`, `df.iloc[:end]`) rather than reimplementing the t-test/KS-test logic â€” same reuse pattern as `forecaster/model_tools.py::rolling_origin_backtest` reusing `fit_ets`/`fit_sarima`/`fit_gradient_boosted_trees`. Unlike `rolling_origin_backtest`, this is genuinely CHEAP per check (a t-test and KS test over numpy arrays, no model fitting) â€” the docstring says so explicitly, and `n_checks` defaults higher-effort-tolerant than `rolling_origin_backtest`'s `n_origins` as a result. `step_size` defaults to `recent_window_size` (fully non-overlapping recent windows across checks, mirroring `rolling_origin_backtest`'s own non-overlapping-origins default) but is independently configurable, unlike `rolling_origin_backtest` which has no equivalent knob. `persistence_threshold_frac` (default 0.5) is an explicit parameter, not a hidden constant â€” same "thresholds are parameters" convention `recommend_retraining` already documents in its own docstring. Checks are reported chronologically (oldest-to-newest), reversed from the walk-backward computation order, same convention as `rolling_origin_backtest`'s `origins.reverse()`.
- `retrain/retrain_tools.py`'s `compare_candidate_to_deployed` requires a relative improvement above `improvement_threshold_pct` (default 10%) before recommending a swap, not just any improvement â€” this is deliberate, to avoid redeploying (and resetting the monitoring baseline) over noise-level metric differences between two similar fits.
- `retrain/retrain_tools.py`'s `execute_redeploy` imports `deploy/forecast_tools.py` as a whole module, so it needs the `deploy` extra (statsmodels + scikit-learn) installed regardless of `model_type` â€” even `model_type="naive"` triggers the same module-level import. The `retrain` extra itself stays dependency-free on purpose, since the diagnostic tools don't need it; only `execute_redeploy` does.
- `analyst/analysis_tools.py`'s `check_stationarity` runs ADF AND KPSS together and reports an effect size for each. **Field names changed when KPSS was added**: the ADF fields were renamed from `p_value`/`is_likely_stationary` to `adf_p_value`/`adf_is_likely_stationary`, so two different tests' results in the same dict wouldn't be ambiguous. If you're referencing this tool's output anywhere (a `SKILL.md`, a test, an integration), use the prefixed names.
  - ADF's effect size (`mean_reversion_lambda`, `mean_reversion_half_life_periods`) comes from an OLS fit of `delta_y_t = lambda*y_{t-1} + mu + epsilon_t` via `statsmodels.regression.linear_model.OLS` (not derived from `adfuller`'s own internals, so it stays interpretable on its own and isn't coupled to ADF's autolag choice; switched from raw `np.linalg.lstsq` specifically to get `.conf_int()` for free). It also reports a confidence interval for both lambda and half-life (`mean_reversion_lambda_ci_lower/upper`, `mean_reversion_half_life_ci_lower/upper`). **Don't assume `mean_reversion_lambda >= 0` under a true unit root** â€” the OLS estimate is well-known to skew slightly negative in finite samples even with zero true reversion (exactly why ADF needs its own non-standard critical values rather than a plain t-test); this bit a test during development (see `test_check_stationarity_reports_weak_reversion_for_a_pure_random_walk` in `tests/test_analyst_tools.py`, which asserts on `adf_is_likely_stationary`/half-life magnitude instead of the sign of lambda for that reason). **The half-life CI mapping is NOT symmetric/naive**: half-life is an INCREASING function of lambda on lambda < 0 (more negative lambda = faster reversion = shorter half-life), so `half_life_ci_lower` comes from `lambda_ci_lower` and `half_life_ci_upper` comes from `lambda_ci_upper`, in that same order (not swapped). `half_life_ci_upper` is `None` whenever `lambda_ci_upper >= 0` -- lambda's CI reaching non-negative territory means the data can't rule out arbitrarily slow reversion, so the upper bound is genuinely unbounded, not just "a large number that got skipped." Don't "fix" a `None` here by substituting a large finite placeholder.
  - KPSS's effect size (`kpss_effect_size`) is the KPSS statistic as a multiple of its own 5% critical value, computed from values `kpss()` already returns (no re-implementation of its Newey-West long-run variance estimator). This exists specifically because KPSS's own p-value is coarse: `statsmodels` interpolates it from just four lookup-table points and clips it at the table's edges, so p-values of 0.01 or 0.10 can mean "barely past the boundary" or "wildly past it" with no way to tell from the p-value alone. `_kpss_effect_size` deliberately suppresses the `InterpolationWarning` `kpss()` raises at those clipped boundaries â€” that's not silencing a problem, the effect size *is* the more informative answer to the exact thing the warning is pointing at.
  - ADF and KPSS have opposite null hypotheses (ADF: unit root; KPSS: stationary) â€” `_combined_stationarity_interpretation` encodes the standard four-way readout (both agree stationary, both agree non-stationary, or either disagreement direction, each with its own guidance e.g. "try `kpss_regression='ct'`"). Keep new stationarity-adjacent tools consistent with this framing rather than reporting ADF and KPSS as two independent, unrelated facts.
- `analyst/analysis_tools.py`'s `basic_stats` reports a confidence interval for the mean (`mean_ci_lower`/`mean_ci_upper`, Student's t via `scipy.stats.t`, not the normal approximation, since sample sizes here can be small) â€” both `None` for a constant series (zero variance). Uses `values.count()` (non-null count) for the sample size in the standard-error calculation, NOT `len(df)` -- those differ whenever there are missing values, and using `len(df)` would understate the standard error (and thus the CI width) by inflating the effective sample size.
- `analyst/analysis_tools.py`'s `acf_pacf_summary` and `detect_anomalies_zscore` both report an effect size for anything they flag, sorted strongest/most-extreme first (not chronologically) before capping the list for brevity. **This changed both fields' shape, not just added new ones**: `significant_acf_lags` went from a bare list of lag integers to a list of `{lag, acf, ci_lower, ci_upper, effect_size}` dicts; `detect_anomalies_zscore` replaced `anomaly_dates` (date strings only) with `anomalies` (`{date, value, z_score}` dicts) plus a new `max_abs_z_score` summary field. Same rationale as the ADF/KPSS effect sizes above: a bare "flagged" boolean throws away magnitude the function already computed internally to make that decision â€” don't add a new flagging tool that collapses a computed score down to a boolean without also surfacing the score.
- `analyst/analysis_tools.py`'s `acf_pacf_summary` significance test is a REAL correctness fix, not just an added field: it now uses `statsmodels.tsa.stattools.acf(..., alpha=...)`'s per-lag Bartlett-formula confidence intervals instead of a single global `1.96/sqrt(n)` threshold. The Bartlett standard error grows with lag (`varacf[2:] *= 1 + 2*cumsum(acf[1:-1]**2)` inside `statsmodels`) because it's derived as if the series were an MA(k-1) at lag k -- a uniform threshold is only actually correct at lag 1 and understates the true threshold everywhere past it. This has a real, demonstrated effect: on the project's synthetic series, the old uniform threshold ranked lag 7 as the single strongest significant lag; the corrected per-lag threshold ranks lag 1 first instead (its Bartlett SE is the tightest of any lag, since no prior lags inflate it). `significance_threshold` (a single float) was renamed `significance_alpha` (the alpha level fed to `acf()`, e.g. 0.05) since there's no longer one global threshold to report. `acf(..., alpha=...)` returns confidence intervals CENTERED ON THE ESTIMATED ACF VALUE, not on zero (`statsmodels`' own docs call this out explicitly, since it differs from `plot_acf`) -- significance here means the interval excludes zero, i.e. `abs(acf_vals[i]) > half_width`, not `abs(acf_vals[i]) > ci_upper` or similar. Don't compute the significance check by comparing against the raw `confint` bounds directly.
- `analyst/analysis_tools.py`'s `detect_seasonality_period` runs Fisher's g-test for hidden periodicity on the FULL periodogram (all non-zero Fourier frequencies, `q = len(freqs)`), THEN separately filters to `[min_period, max_period]` only for which candidates get REPORTED in `top_candidate_periods` -- don't collapse these into one step. The significance test's null distribution depends on `q` being the true, complete set of frequencies; filtering before computing `g0`/`p_value` would silently invalidate the test. The p-value formula (`p_value = min(1.0, q * (1 - g0) ** (q - 1))`) is the standard CONSERVATIVE upper-bound approximation (the leading term of Fisher's full alternating-series formula, same approximation the GeneCycle R package's `fisher.g.test` uses) -- not the exact formula. Don't "improve precision" by implementing the full alternating series without checking its numerical stability first (large `q` with small `g0` can require many terms and risks overflow in the binomial-coefficient terms).
- `analyst/analysis_tools.py`'s `detect_anomalies_robust_zscore` and its `_rolling_mad` helper rely on `series.rolling(window=..., min_periods=..., center=...).median()` and a separate `.rolling(...).apply(mad, raw=True)` call USING IDENTICAL windowing parameters to both. This is required for correctness, not just style: the modified z-score formula needs each point's local MAD computed using THAT SAME window's own median as the reference (not some other window's median) -- the two rolling calls only agree on medians point-for-point because their `window`/`min_periods`/`center` arguments match exactly. If you change one, change the other the same way.
- `analyst/analysis_tools.py`'s `detect_changepoints` uses a PERMUTATION TEST (`n_permutations`, default 500, seeded via `seed`, default 42) for its CUSUM statistic's p-value rather than a closed-form asymptotic distribution -- same reasoning `deploy/forecast_tools.py`'s `forecast_ets` already uses simulation (`n_simulations`) for its prediction intervals instead of a closed-form formula: a permutation test is exact/nonparametric by construction and sidesteps needing to derive or hard-code asymptotic critical values correctly. **Binary segmentation's per-split p-values are NOT a global significance guarantee for the full set of reported changepoints** -- each split's test is local to whatever segment existed at that point in the recursion. This is a standard, well-documented limitation of binary segmentation as a class of algorithm (not unique to this implementation, not a bug) -- don't present `detect_changepoints`' results as having an exact family-wise error rate, and don't "fix" this by adding a post-hoc correction without understanding why one isn't already there. `detect_changepoints` answers a different question from `detect_anomalies_zscore`/`detect_anomalies_robust_zscore`: a lasting shift in MEAN LEVEL vs. an individual point outlier -- a single huge isolated spike can trip an anomaly detector without being a real changepoint, and a modest sustained level shift can be a real changepoint without tripping either anomaly detector.
- `forecaster/model_tools.py`'s `fit_naive_baselines`/`fit_ets`/`fit_sarima`/`fit_gradient_boosted_trees` all now return `backtest_metrics` via `compute_metrics_with_ci` (bootstrap CI, percentile method, `n_bootstrap`/`confidence_level`/`seed` params, deterministic given the same seed) instead of the bare `compute_metrics`, plus new `holdout_actuals`/`holdout_predicted` fields (used by `diebold_mariano_test`, see below). `residual_diagnostics` (ETS/SARIMA) also gained `ljung_box_statistic`/`ljung_box_critical_value`/`ljung_box_effect_size`, same "how far past the boundary, not just past/fail" pattern as `ts-analyst`'s KPSS/ACF effect sizes. **This is additive, not a rename** â€” unlike the ADF/KPSS and ACF changes in `analyst/analysis_tools.py`, all pre-existing field names (`mae`, `rmse`, `mape_pct`, `ljung_box_p_value`, etc.) are unchanged; only new keys were added.
- `forecaster/model_tools.py`'s `diebold_mariano_test` is a genuinely new capability, not an enrichment of an existing tool: a Diebold & Mariano (1995)-style paired comparison of two models' forecast errors on the SAME holdout, using a Newey-West/Bartlett-kernel HAC-robust variance estimate of the mean loss differential. **Read the docstring's math carefully before touching this function** â€” an earlier draft defaulted the autocovariance truncation lag to `horizon = n` (treating the whole holdout as a single classic-DM "h-step-ahead forecast horizon"), which is degenerate: it collapses the intended Harvey-Leybourne-Newbold small-sample correction factor to exactly zero when `h = n` (verified: `(n + 1 - 2h + h(h-1)/n)/n = 0` algebraically when `h = n`), making every test result non-significant regardless of the actual data. The shipped version instead uses `n_lags` (an ordinary Newey-West lag-selection parameter, defaulting to the standard Newey & West 1994 automatic rule `floor(4*(n/100)^(2/9))`, decoupled from any "forecast horizon" concept) and a plain Student's t(df=n-1) reference distribution for small-sample conservatism, rather than the classic HLN correction formula specifically â€” a more conservative, defensible choice given this toolkit's fixed-single-origin backtest structure doesn't actually match the repeated-rolling-origin setup the classic HLN formula was derived for. Pass `n_lags=0` only when BOTH models being compared are one-step-ahead backtests (independent errors expected); the default assumes genuinely autocorrelated multi-step forecast errors, appropriate for `fit_ets`/`fit_sarima`. Comparing across the one-step-ahead/multi-step boundary (e.g. `fit_gradient_boosted_trees` vs. `fit_sarima`) with this test answers "do the error numbers differ significantly," NOT "is this a fair comparison" â€” the evaluation-setting mismatch documented on `fit_gradient_boosted_trees` itself still applies regardless of what this test says.
- `forecaster/model_tools.py`'s `fit_ets`/`fit_sarima` gained `aicc` (via `_aicc`, Hurvich & Tsai 1989 small-sample AIC correction -- `None` when `n - k - 1 <= 0`, i.e. too few training observations relative to `k = len(fit.params)` for the correction to be defined) and `backtest_interval_coverage` (via `_interval_coverage`). `_interval_coverage` deliberately mirrors `monitor/monitor_tools.py::compare_forecast_to_actuals`'s `interval_coverage` shape AND its 15-percentage-point `well_calibrated` threshold exactly, on purpose â€” same check, run one layer earlier (during backtesting, not after deployment). `fit_ets`'s interval is SIMULATED (`fit.simulate`, same approach as `deploy/forecast_tools.py::forecast_ets`, wrapped in try/except since simulation can fail for some parameter combinations â€” `backtest_interval_coverage` reports `{"error": ...}` in that case rather than omitting the field); `fit_sarima`'s is ANALYTIC (`prediction.conf_int()`, always available, no failure case). `fit_gradient_boosted_trees` does NOT get this field â€” it has no native prediction interval, same limitation as `forecast_gradient_boosted_trees` in `ts-deploy`.
- `forecaster/model_tools.py`'s `rolling_origin_backtest` reuses `fit_ets`/`fit_sarima`/`fit_gradient_boosted_trees` directly rather than reimplementing model fitting: it calls the SAME `fit_*` function repeatedly on progressively larger PREFIXES of `df` (`df.iloc[:test_end]`), relying on the fact that each `fit_*` function already does `train_test_split(df, holdout_size)` internally (last `holdout_size` rows = test, everything before = train) -- so truncating the input df to end at each origin automatically gives an expanding-window origin scheme for free, no separate windowing logic needed. `model_type` dispatch (`"ets"`/`"sarima"`/`"gbt"` -> the matching `fit_*` function, via `_ROLLING_ORIGIN_FIT_FUNCTIONS`) deliberately mirrors `retrain/retrain_tools.py::execute_redeploy`'s existing `model_type`/`params` dispatch convention for consistency across the codebase -- if you add a new `model_type` string to one, consider whether the other should know about it too. `"naive"` is deliberately NOT supported here (see the function's own docstring for why) and `fit_naive_baselines`' different two-baseline return shape isn't compatible with this function's single-result-per-origin assumption anyway.
- `forecaster/model_tools.py`'s `search_sarima_orders` is explicitly scoped as advisory, not authoritative â€” this was a deliberate design tension flagged before implementation: this toolkit's whole design (see `ts-forecaster/SKILL.md`) asks the agent to reason about model settings from `ts-analyst`'s findings, not grid-search blindly. Don't strengthen this tool's own claims (e.g. auto-selecting and returning a single "best" order rather than a `top_candidates` shortlist) without discussing that tension explicitly â€” it would work against the project's existing design principle, not just add a feature. `d`/`seasonal_d` are intentionally NOT searched (held fixed, caller-supplied) for the same reason. Verified directly that the numerically-best-AICc candidate on the project's own synthetic data has `residuals_look_like_white_noise: false` â€” a live, concrete example of why "ranked first by AICc" isn't the same as "actually good," baked into the tool's own docstring and `interpretation` field, not just asserted.
- `deploy/forecast_tools.py`'s `forecast_gradient_boosted_trees` now fits THREE `GradientBoostingRegressor` models per call, not one â€” the original point model, plus a `loss="quantile"` model at `alpha/2` and another at `1 - alpha/2`, giving GBT a prediction interval for the first time (previously the only one of the four `forecast_*` tools with none, ever). This triples the tool's compute cost â€” a deliberate tradeoff, not an oversight. The recursive loop still follows only the POINT model's own predictions as lag features for later steps; the quantile models are evaluated at each step against that same point-derived feature row rather than recursing on their own predictions, specifically to avoid three diverging trajectories. Independently-fit quantile regressors CAN cross (`lower_pred > upper_pred`) especially on limited data â€” guarded with `np.minimum`/`np.maximum` on the two arrays before returning, not by refitting or discarding. This interval is explicitly weaker than SARIMA's/ETS's: it doesn't grow with the recursive compounding-error risk that's this tool's headline caveat, so don't let `interval_note`'s mere presence imply the same rigor â€” both `interval_note` and `caveat` say this explicitly, keep both if you touch this function.
- `deploy/forecast_tools.py` gained `_aicc`, duplicated from `forecaster/model_tools.py::_aicc` rather than imported cross-layer (same precedent as `_build_lag_features`'s existing duplication, documented in its own docstring) â€” `forecast_ets`/`forecast_sarima` now report `aicc` alongside `aic`. This value is NOT directly comparable to `ts-forecaster`'s `fit_ets`/`fit_sarima` `aicc` for the same series â€” Layer 2 computes it on the training split only (`n` = `len(train)`), Layer 3 refits on the FULL series (`n` = `len(df)`), so both `n` and the fitted params can differ. Its main use here is as a diagnostic on the deployment refit itself, not a rematch against Layer 2's candidate comparison.
- `deploy/forecast_tools.py`'s `_check_forecast_plausibility` (embedded as `plausibility_check` in every `forecast_*` tool's result, including `forecast_ensemble`) automates part of `ts-deploy/SKILL.md` Step 3's "does this look plausible" eyeball check. **This is deliberately NOT a hypothesis test** â€” there's no null distribution being tested, only an empirical comparison ("has the series done a move like this before"), so don't describe its `endpoint_change_z_score` as a p-value-bearing statistic in docs or output. It compares the forecast's ENDPOINT change (last forecast value minus last observed value) against the empirical distribution of `horizon`-length changes taken from the historical series itself (`values[horizon:] - values[:-horizon]`) â€” not against a fitted parametric distribution â€” and separately flags whether the forecast's own min/max ever leaves the historical min/max range. Returns `None` for the z-score/percentile fields (not zero, not an error) when `n <= horizon` (not enough history) or when historical change variance is ~zero â€” both real "can't compute this" states, not degenerate results to paper over. `is_extreme_relative_to_history` uses `abs(z) > 2.5` as its threshold â€” a fixed, undocumented-elsewhere constant; if you change it, note why here.
- `deploy/forecast_tools.py`'s `_future_dates` has a real correctness fix: `pd.infer_freq` does NOT return `None` for a series with fewer than 3 dates the way the function's original comment assumed â€” it RAISES `ValueError`. The pre-existing `freq = pd.infer_freq(df["date"]) or "D"` fallback only ever covered the "inferred an ambiguous/no frequency" case, not the "too few dates to even attempt inference" case, so any `forecast_*` call on a 1-2 row series crashed instead of falling back to daily. Found via `forecast_naive`'s new short-series edge-case tests (see below), not via any user report â€” wrap `pd.infer_freq` in `try/except ValueError` rather than relying on `or "D"` alone if you touch this function again.
- `deploy/forecast_tools.py`'s `forecast_naive` (via `_fit_naive_values`) now has an analytic prediction interval â€” previously the only `forecast_*` tool with NONE, ever. Uses the naive method's own in-sample residuals as the basis (flat naive: `np.diff(values)`, i.e. one-step differences, since a flat naive forecast IS a random-walk-with-no-drift model; seasonal naive: `values[m:] - values[:-m]`, seasonal differences) and the standard textbook formula (Hyndman & Athanasopoulos) `sigma_h = sigma * sqrt(cycles)` where `cycles = h` for flat naive and `cycles = floor((h-1)/m) + 1` for seasonal naive â€” the seasonal interval only widens at each COMPLETE additional seasonal cycle (steps within one cycle share the same uncertainty), not at every single step, which is why it's `floor((h-1)/m) + 1` rather than a per-step `sqrt(h)`. `z` comes from `scipy.stats.norm.ppf` (a new import in this module), NOT from a t-distribution â€” the residual count here can be large (the full historical series), so the normal approximation is standard and matches the textbook source, unlike `basic_stats`' small-sample Student's t elsewhere in this toolkit. Falls back to point-forecast-only (`lower`/`upper` both `None`, not zero-width) when fewer than 2 residuals are available to compute a standard deviation from â€” a real "can't compute this" state, same pattern as `_check_forecast_plausibility`'s own `None` fields for insufficient history, not an error to paper over. `forecast_ensemble`'s `"naive"` branch was updated to actually use this interval (previously always passed `lower=None, upper=None` for naive unconditionally) â€” combinations including `"naive"` can now get a combined interval where before they categorically couldn't.
- `deploy/forecast_tools.py`'s `forecast_ensemble` fills the gap flagged as a Next Step in the `ts-forecaster` blog post ("no ensemble or model-averaging tooling yet... arguably `ts-deploy`'s question to answer") â€” combining backtest-validated candidates is this layer's job (producing a forecast to use), not `ts-forecaster`'s (evaluating candidates individually). Internally calls the private `_fit_*_values` raw helpers (`_fit_naive_values`, `_fit_ets_values`, `_fit_sarima_values`, `_fit_gbt_values`) directly rather than the public `forecast_*` functions â€” this is required, not a style choice: the public functions run their output through `_format_forecast`'s truncation (`cap=60`), which for `horizon > 60` drops the middle of the array entirely rather than keeping it, so combining already-truncated component results would silently misalign or shrink the weighted average. The four public `forecast_*` functions were refactored into thin wrappers around these same raw helpers specifically so `forecast_ensemble` could reuse the untruncated arrays without duplicating model-fitting logic. `weights` accepts un-normalized values on purpose (normalizes internally by dividing by `sum(weights)`) specifically so raw inverse-error values from a Layer 2 comparison can be passed straight through. The combined interval is reported ONLY when every included model contributes one of its own (checked via `all(iv is not None for iv in interval_pairs)`) â€” deliberately not silently dropping a no-interval model's weight from a partial interval average, which would misrepresent the combination as more rigorous than it is; `interval_note` explains which model(s) were missing one when this happens.
- `deploy/forecast_tools.py`'s `forecast_ensemble` combined interval is a VARIANCE combination, not a bound average â€” this REPLACED the original implementation (which literally did `weighted_lower = sum(w_i * lo_i)`, `weighted_upper = sum(w_i * up_i)`), after the user flagged that approach as one of two remaining polish items worth fixing. Current approach: each component's own interval half-width becomes an implied sigma (`sigma_i = (up_i - lo_i) / (2*z)`, `z` from `scipy.stats.norm.ppf` at the shared `confidence_level`), combined as `combined_sigma = sqrt(sum(w_i^2 * sigma_i^2))` under an INDEPENDENCE assumption between components, then the final interval is rebuilt symmetrically around `weighted_point` as `weighted_point Â± z * combined_sigma`. **This can produce a NARROWER combined interval than any single component's own interval** â€” verified directly: two identical naive components at equal weight (0.5/0.5) combine to exactly `1/sqrt(2)` (â‰ˆ0.707x) the width of either alone, since `sqrt(0.5^2 + 0.5^2) = 1/sqrt(2)`. This is the correct, expected mathematical consequence of combining independent estimates, NOT a bug to "fix" by clamping the combined width to be at least as wide as the widest component â€” if you're tempted to add that clamp, don't; it would misrepresent the (still real, still documented as optimistic) independence assumption as something else entirely. The independence assumption itself IS the acknowledged weakness here (models fit on the same series share real error structure, so true combined uncertainty is wider than this formula gives) â€” `interval_note` and this codebase's docstrings both frame the result as a lower bound, not a precise value; don't strengthen that framing without addressing the underlying assumption.
- `deploy/forecast_tools.py`'s `_fit_gbt_values`/`forecast_gradient_boosted_trees` gained a bootstrap confidence interval on `feature_importances` â€” the other of the two polish items from the same round. **This is a SHAPE CHANGE**: `feature_importances` went from `{col: float}` to `{col: {importance, ci_lower, ci_upper}}`, same "changed shape, not just added fields" pattern as `ts-analyst`'s earlier ACF/anomaly-detector upgrades (see above) â€” if you're reading this field anywhere, update for the nested shape. The bootstrap resamples TRAINING ROWS (`feat[feature_cols].iloc[idx]`/`feat["value"].iloc[idx]` via `rng.integers`), not resampled errors like `compute_metrics_with_ci` elsewhere in this toolkit â€” there's no "error" to resample for a feature-importance question, only what the model was fit ON. This is genuinely expensive, unlike the rest of this toolkit's bootstrap CIs (which resample a precomputed metric formula in pure numpy): each of `n_bootstrap` (default 100) resamples is a FULL `GradientBoostingRegressor` refit at the same `n_estimators`/`max_depth`/`learning_rate` as the point model, on top of the three model fits `_fit_gbt_values` already does for the point forecast + its own quantile interval â€” so a default call now costs roughly `3 + n_bootstrap` total model fits, not 3. `n_bootstrap=0` is a first-class, documented opt-out (skips the loop entirely, returns `ci_lower`/`ci_upper` as `None`, not zero-width) â€” `forecast_ensemble`'s internal GBT branch always passes `n_bootstrap=0` explicitly, since ensemble results never surface `feature_importances` at all and paying for the CI there would be pure waste; don't remove that `n_bootstrap=0` from the ensemble call without understanding why it's there. Confirmed on the project's synthetic data that the CI is actually informative: `lag_7` (the real weekly seasonality) showed `importance=0.593` but a CI of `[0.26, 0.77]`, since `lag_7` and `lag_14` compete to explain the same weekly pattern and which one "wins" varies by resample â€” not a decorative addition.
