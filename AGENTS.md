# Omen

Five layers of agentic time series tooling. Each layer is a FastMCP server
(typed tools) plus a companion OpenClaw skill (markdown playbook telling an
agent how to sequence those tools and what to report). Built to be driven
by OpenClaw + an LLM agent (developed against GLM-5.2 on Ollama Cloud, but
nothing is model-specific).

- `analyst`    -> `ts-analyst`    : explore a series, recommend an approach. No fitting.
- `forecaster` -> `ts-forecaster` : fit + backtest candidate models on a held-out window.
- `deploy`     -> `ts-deploy`     : retrain the chosen model on full data, forecast forward.
- `monitor`    -> `ts-monitor`    : compare deployed forecast vs. reality, detect drift, recommend retraining.
- `retrain`    -> `ts-retrain`    : on retrain_now, re-run analyst+forecaster and decide deterministically whether the new candidate beats what's deployed; redeploys only through the gated `execute_redeploy(confirmed=True)` tool, either human-approved (default) or via an explicitly authorized autonomous mode.

Full architecture/setup is in README.md — don't duplicate it here.

**Project naming**: the project/package/repo directory is `omen` (formerly
`agentic-ts-toolkit` — renamed, not rebranded from scratch). The five
layers deliberately KEPT their existing `ts-analyst`/`ts-forecaster`/
`ts-deploy`/`ts-monitor`/`ts-retrain` server names, skill names, and
MCP-qualified tool names (e.g. `ts-analyst__basic_stats`) unchanged —
that's an already-established, separately-branded layer referenced
everywhere (every `SKILL.md`, every test, every docstring, every blog
post), and renaming it alongside the project name would have been a much
larger, more disruptive change than what was asked for. If you're adding
new tools, keep using the `ts-*` naming convention for them; don't start
mixing in `omen-*`-prefixed names for consistency with the project name,
since that would just create two competing conventions.

## Commands

```bash
pip install -e ".[all,dev]"    # everything, including test deps
pytest                          # run the suite
ts-analyst-server                # run a layer's MCP server standalone (Ctrl+C to stop)
python -m build                 # build sdist/wheel for publishing
```

Per-layer installs: `.[analyst]`, `.[forecaster]`, `.[deploy]`, `.[monitor]`, `.[retrain]` if you don't want every dependency.

## Non-obvious conventions

- **One shared `data_prep.py`** at the package root (`omen.data_prep`). Every subpackage imports it — never re-add a per-layer copy.
- **Within a subpackage, tools return plain functions taking a DataFrame; `server.py` wraps them as `@mcp.tool()` and does the CSV loading.** Keep that split — it's what makes the tool logic unit-testable without spinning up MCP (see `tests/`, which import the tools modules directly, not through `server.py`).
- **`recommend_retraining` in `monitor_tools.py` is deliberately deterministic/rule-based**, not an LLM judgment call — that's intentional, not a missed opportunity to "let the agent decide." Keep new decision-style tools (accept/reject, threshold-based verdicts) rule-based for the same reason: reproducibility given the same inputs. `retrain_tools.py`'s `compare_candidate_to_deployed` follows the same rule.
- **Every `SKILL.md` references its tools by their MCP-qualified name** (`ts-analyst__basic_stats`, etc.), not by file path — these are OpenClaw skills, not exec-based scripts. If you add a tool, update the matching `SKILL.md`'s tool list too.
- **Tests use `pytest.importorskip("statsmodels")` / `"sklearn"`** in the layers that need them (`analyst`, `forecaster`, `deploy`), so `pip install -e .` (core deps only) doesn't break the whole suite. `monitor` and `data_prep` tests have no such guard — keep it that way, they're meant to run everywhere. `retrain` is mostly in that camp too (`record_deployment`/`load_deployment_manifest`/`compare_candidate_to_deployed` need nothing beyond core), but `test_retrain_tools.py`'s one test of `execute_redeploy`'s real dispatch path uses a per-test (not module-level) `importorskip`, since `execute_redeploy` alone needs the `deploy` extra — don't move that guard to the top of the file, it would skip tests that don't need it.
- **`ts-retrain` is the one place in the toolkit with durable state** — `record_deployment`/`load_deployment_manifest` read and write a JSON manifest (`deployment_manifest.json` by default, next to the series CSV; see `_manifest_path_for` in `retrain_tools.py`). Every other tool in every other layer is a pure function of its explicit inputs — don't add more file-based state elsewhere without a similarly strong reason.
- **`ts-retrain` only ever changes a deployment through `execute_redeploy`, which refuses to run without `confirmed=True`** (no default performs the action). Two paths reach that confirmation, both documented in `SKILL.md`: human approval in-conversation (default), or an explicitly authorized autonomous mode where the skill calls `execute_redeploy(confirmed=True)` itself. The authorization check for autonomous mode is a prose contract enforced by the skill's instructions, not something the code validates — `execute_redeploy` itself doesn't know or care which path led to `confirmed=True`. Don't weaken the skill's "default to human mode when ambiguous" language, and don't add a code path that sets `confirmed=True` by default anywhere.
  - Human-confirmed example: after `ts-retrain` reports `should_redeploy: true` and stops, the user says *"Go ahead and redeploy the candidate you just recommended"* — that follow-up is what authorizes the `execute_redeploy(..., confirmed=True)` call.
  - Autonomous-mode example: authorization is scoped per series, stated explicitly, e.g. *"For the `daily_demand.csv` series specifically, you're authorized to redeploy automatically whenever ts-retrain finds a candidate that beats the current deployment — no need to ask me first. Everything else still needs my confirmation as usual."* Absent wording this explicit and this scoped, treat autonomous mode as not authorized.
- **Both `SARIMAX(...)` call sites (`forecaster/model_tools.py::fit_sarima`, `deploy/forecast_tools.py::forecast_sarima`) pass a single-column DataFrame (`df[["value"]]`), not a Series (`df["value"]`).** This isn't stylistic -- statsmodels does an in-place `ndarray.shape=` reshape internally, but only for 1-D endog, and NumPy 2.5+ deprecated in-place shape mutation, so a Series triggers a `DeprecationWarning` on every fit. A single-column DataFrame is already 2-D, so statsmodels skips that code path -- verified identical AIC/BIC, residuals, and forecast values/shapes either way. If you add another `SARIMAX(...)` call site, pass a DataFrame the same way.

## Known modeling caveats (intentional, not bugs)

- `forecaster/model_tools.py`'s gradient-boosted-trees backtest is **one-step-ahead** (true lagged values); `deploy/forecast_tools.py`'s gradient-boosted-trees forecast is **recursive** (predictions feed back in as lags). These are not directly comparable to each other or to ETS/SARIMA's genuine multi-step forecasts — both modules' docstrings and return values flag this explicitly. Don't "fix" this asymmetry without understanding why it exists.
- `monitor/monitor_tools.py`'s `detect_data_drift` cannot distinguish an ongoing trend/seasonal transition from a genuine regime change — confirmed empirically that it flags `drift_detected=True` on trending synthetic data with zero injected anomaly. This is by design (a coarse, fast check); read `interpretation`, don't treat the boolean as ground truth.
- `retrain/retrain_tools.py`'s `compare_candidate_to_deployed` requires a relative improvement above `improvement_threshold_pct` (default 10%) before recommending a swap, not just any improvement — this is deliberate, to avoid redeploying (and resetting the monitoring baseline) over noise-level metric differences between two similar fits.
- `retrain/retrain_tools.py`'s `execute_redeploy` imports `deploy/forecast_tools.py` as a whole module, so it needs the `deploy` extra (statsmodels + scikit-learn) installed regardless of `model_type` — even `model_type="naive"` triggers the same module-level import. The `retrain` extra itself stays dependency-free on purpose, since the diagnostic tools don't need it; only `execute_redeploy` does.
- `analyst/analysis_tools.py`'s `check_stationarity` runs ADF AND KPSS together and reports an effect size for each. **Field names changed when KPSS was added**: the ADF fields were renamed from `p_value`/`is_likely_stationary` to `adf_p_value`/`adf_is_likely_stationary`, so two different tests' results in the same dict wouldn't be ambiguous. If you're referencing this tool's output anywhere (a `SKILL.md`, a test, an integration), use the prefixed names.
  - ADF's effect size (`mean_reversion_lambda`, `mean_reversion_half_life_periods`) comes from an OLS fit of `delta_y_t = lambda*y_{t-1} + mu + epsilon_t` via `statsmodels.regression.linear_model.OLS` (not derived from `adfuller`'s own internals, so it stays interpretable on its own and isn't coupled to ADF's autolag choice; switched from raw `np.linalg.lstsq` specifically to get `.conf_int()` for free). It also reports a confidence interval for both lambda and half-life (`mean_reversion_lambda_ci_lower/upper`, `mean_reversion_half_life_ci_lower/upper`). **Don't assume `mean_reversion_lambda >= 0` under a true unit root** — the OLS estimate is well-known to skew slightly negative in finite samples even with zero true reversion (exactly why ADF needs its own non-standard critical values rather than a plain t-test); this bit a test during development (see `test_check_stationarity_reports_weak_reversion_for_a_pure_random_walk` in `tests/test_analyst_tools.py`, which asserts on `adf_is_likely_stationary`/half-life magnitude instead of the sign of lambda for that reason). **The half-life CI mapping is NOT symmetric/naive**: half-life is an INCREASING function of lambda on lambda < 0 (more negative lambda = faster reversion = shorter half-life), so `half_life_ci_lower` comes from `lambda_ci_lower` and `half_life_ci_upper` comes from `lambda_ci_upper`, in that same order (not swapped). `half_life_ci_upper` is `None` whenever `lambda_ci_upper >= 0` -- lambda's CI reaching non-negative territory means the data can't rule out arbitrarily slow reversion, so the upper bound is genuinely unbounded, not just "a large number that got skipped." Don't "fix" a `None` here by substituting a large finite placeholder.
  - KPSS's effect size (`kpss_effect_size`) is the KPSS statistic as a multiple of its own 5% critical value, computed from values `kpss()` already returns (no re-implementation of its Newey-West long-run variance estimator). This exists specifically because KPSS's own p-value is coarse: `statsmodels` interpolates it from just four lookup-table points and clips it at the table's edges, so p-values of 0.01 or 0.10 can mean "barely past the boundary" or "wildly past it" with no way to tell from the p-value alone. `_kpss_effect_size` deliberately suppresses the `InterpolationWarning` `kpss()` raises at those clipped boundaries — that's not silencing a problem, the effect size *is* the more informative answer to the exact thing the warning is pointing at.
  - ADF and KPSS have opposite null hypotheses (ADF: unit root; KPSS: stationary) — `_combined_stationarity_interpretation` encodes the standard four-way readout (both agree stationary, both agree non-stationary, or either disagreement direction, each with its own guidance e.g. "try `kpss_regression='ct'`"). Keep new stationarity-adjacent tools consistent with this framing rather than reporting ADF and KPSS as two independent, unrelated facts.
- `analyst/analysis_tools.py`'s `basic_stats` reports a confidence interval for the mean (`mean_ci_lower`/`mean_ci_upper`, Student's t via `scipy.stats.t`, not the normal approximation, since sample sizes here can be small) — both `None` for a constant series (zero variance). Uses `values.count()` (non-null count) for the sample size in the standard-error calculation, NOT `len(df)` -- those differ whenever there are missing values, and using `len(df)` would understate the standard error (and thus the CI width) by inflating the effective sample size.
- `analyst/analysis_tools.py`'s `acf_pacf_summary` and `detect_anomalies_zscore` both report an effect size for anything they flag, sorted strongest/most-extreme first (not chronologically) before capping the list for brevity. **This changed both fields' shape, not just added new ones**: `significant_acf_lags` went from a bare list of lag integers to a list of `{lag, acf, ci_lower, ci_upper, effect_size}` dicts; `detect_anomalies_zscore` replaced `anomaly_dates` (date strings only) with `anomalies` (`{date, value, z_score}` dicts) plus a new `max_abs_z_score` summary field. Same rationale as the ADF/KPSS effect sizes above: a bare "flagged" boolean throws away magnitude the function already computed internally to make that decision — don't add a new flagging tool that collapses a computed score down to a boolean without also surfacing the score.
- `analyst/analysis_tools.py`'s `acf_pacf_summary` significance test is a REAL correctness fix, not just an added field: it now uses `statsmodels.tsa.stattools.acf(..., alpha=...)`'s per-lag Bartlett-formula confidence intervals instead of a single global `1.96/sqrt(n)` threshold. The Bartlett standard error grows with lag (`varacf[2:] *= 1 + 2*cumsum(acf[1:-1]**2)` inside `statsmodels`) because it's derived as if the series were an MA(k-1) at lag k -- a uniform threshold is only actually correct at lag 1 and understates the true threshold everywhere past it. This has a real, demonstrated effect: on the project's synthetic series, the old uniform threshold ranked lag 7 as the single strongest significant lag; the corrected per-lag threshold ranks lag 1 first instead (its Bartlett SE is the tightest of any lag, since no prior lags inflate it). `significance_threshold` (a single float) was renamed `significance_alpha` (the alpha level fed to `acf()`, e.g. 0.05) since there's no longer one global threshold to report. `acf(..., alpha=...)` returns confidence intervals CENTERED ON THE ESTIMATED ACF VALUE, not on zero (`statsmodels`' own docs call this out explicitly, since it differs from `plot_acf`) -- significance here means the interval excludes zero, i.e. `abs(acf_vals[i]) > half_width`, not `abs(acf_vals[i]) > ci_upper` or similar. Don't compute the significance check by comparing against the raw `confint` bounds directly.
- `analyst/analysis_tools.py`'s `detect_seasonality_period` runs Fisher's g-test for hidden periodicity on the FULL periodogram (all non-zero Fourier frequencies, `q = len(freqs)`), THEN separately filters to `[min_period, max_period]` only for which candidates get REPORTED in `top_candidate_periods` -- don't collapse these into one step. The significance test's null distribution depends on `q` being the true, complete set of frequencies; filtering before computing `g0`/`p_value` would silently invalidate the test. The p-value formula (`p_value = min(1.0, q * (1 - g0) ** (q - 1))`) is the standard CONSERVATIVE upper-bound approximation (the leading term of Fisher's full alternating-series formula, same approximation the GeneCycle R package's `fisher.g.test` uses) -- not the exact formula. Don't "improve precision" by implementing the full alternating series without checking its numerical stability first (large `q` with small `g0` can require many terms and risks overflow in the binomial-coefficient terms).
- `analyst/analysis_tools.py`'s `detect_anomalies_robust_zscore` and its `_rolling_mad` helper rely on `series.rolling(window=..., min_periods=..., center=...).median()` and a separate `.rolling(...).apply(mad, raw=True)` call USING IDENTICAL windowing parameters to both. This is required for correctness, not just style: the modified z-score formula needs each point's local MAD computed using THAT SAME window's own median as the reference (not some other window's median) -- the two rolling calls only agree on medians point-for-point because their `window`/`min_periods`/`center` arguments match exactly. If you change one, change the other the same way.
- `analyst/analysis_tools.py`'s `detect_changepoints` uses a PERMUTATION TEST (`n_permutations`, default 500, seeded via `seed`, default 42) for its CUSUM statistic's p-value rather than a closed-form asymptotic distribution -- same reasoning `deploy/forecast_tools.py`'s `forecast_ets` already uses simulation (`n_simulations`) for its prediction intervals instead of a closed-form formula: a permutation test is exact/nonparametric by construction and sidesteps needing to derive or hard-code asymptotic critical values correctly. **Binary segmentation's per-split p-values are NOT a global significance guarantee for the full set of reported changepoints** -- each split's test is local to whatever segment existed at that point in the recursion. This is a standard, well-documented limitation of binary segmentation as a class of algorithm (not unique to this implementation, not a bug) -- don't present `detect_changepoints`' results as having an exact family-wise error rate, and don't "fix" this by adding a post-hoc correction without understanding why one isn't already there. `detect_changepoints` answers a different question from `detect_anomalies_zscore`/`detect_anomalies_robust_zscore`: a lasting shift in MEAN LEVEL vs. an individual point outlier -- a single huge isolated spike can trip an anomaly detector without being a real changepoint, and a modest sustained level shift can be a real changepoint without tripping either anomaly detector.
- `forecaster/model_tools.py`'s `fit_naive_baselines`/`fit_ets`/`fit_sarima`/`fit_gradient_boosted_trees` all now return `backtest_metrics` via `compute_metrics_with_ci` (bootstrap CI, percentile method, `n_bootstrap`/`confidence_level`/`seed` params, deterministic given the same seed) instead of the bare `compute_metrics`, plus new `holdout_actuals`/`holdout_predicted` fields (used by `diebold_mariano_test`, see below). `residual_diagnostics` (ETS/SARIMA) also gained `ljung_box_statistic`/`ljung_box_critical_value`/`ljung_box_effect_size`, same "how far past the boundary, not just past/fail" pattern as `ts-analyst`'s KPSS/ACF effect sizes. **This is additive, not a rename** — unlike the ADF/KPSS and ACF changes in `analyst/analysis_tools.py`, all pre-existing field names (`mae`, `rmse`, `mape_pct`, `ljung_box_p_value`, etc.) are unchanged; only new keys were added.
- `forecaster/model_tools.py`'s `diebold_mariano_test` is a genuinely new capability, not an enrichment of an existing tool: a Diebold & Mariano (1995)-style paired comparison of two models' forecast errors on the SAME holdout, using a Newey-West/Bartlett-kernel HAC-robust variance estimate of the mean loss differential. **Read the docstring's math carefully before touching this function** — an earlier draft defaulted the autocovariance truncation lag to `horizon = n` (treating the whole holdout as a single classic-DM "h-step-ahead forecast horizon"), which is degenerate: it collapses the intended Harvey-Leybourne-Newbold small-sample correction factor to exactly zero when `h = n` (verified: `(n + 1 - 2h + h(h-1)/n)/n = 0` algebraically when `h = n`), making every test result non-significant regardless of the actual data. The shipped version instead uses `n_lags` (an ordinary Newey-West lag-selection parameter, defaulting to the standard Newey & West 1994 automatic rule `floor(4*(n/100)^(2/9))`, decoupled from any "forecast horizon" concept) and a plain Student's t(df=n-1) reference distribution for small-sample conservatism, rather than the classic HLN correction formula specifically — a more conservative, defensible choice given this toolkit's fixed-single-origin backtest structure doesn't actually match the repeated-rolling-origin setup the classic HLN formula was derived for. Pass `n_lags=0` only when BOTH models being compared are one-step-ahead backtests (independent errors expected); the default assumes genuinely autocorrelated multi-step forecast errors, appropriate for `fit_ets`/`fit_sarima`. Comparing across the one-step-ahead/multi-step boundary (e.g. `fit_gradient_boosted_trees` vs. `fit_sarima`) with this test answers "do the error numbers differ significantly," NOT "is this a fair comparison" — the evaluation-setting mismatch documented on `fit_gradient_boosted_trees` itself still applies regardless of what this test says.
- `forecaster/model_tools.py`'s `fit_ets`/`fit_sarima` gained `aicc` (via `_aicc`, Hurvich & Tsai 1989 small-sample AIC correction -- `None` when `n - k - 1 <= 0`, i.e. too few training observations relative to `k = len(fit.params)` for the correction to be defined) and `backtest_interval_coverage` (via `_interval_coverage`). `_interval_coverage` deliberately mirrors `monitor/monitor_tools.py::compare_forecast_to_actuals`'s `interval_coverage` shape AND its 15-percentage-point `well_calibrated` threshold exactly, on purpose — same check, run one layer earlier (during backtesting, not after deployment). `fit_ets`'s interval is SIMULATED (`fit.simulate`, same approach as `deploy/forecast_tools.py::forecast_ets`, wrapped in try/except since simulation can fail for some parameter combinations — `backtest_interval_coverage` reports `{"error": ...}` in that case rather than omitting the field); `fit_sarima`'s is ANALYTIC (`prediction.conf_int()`, always available, no failure case). `fit_gradient_boosted_trees` does NOT get this field — it has no native prediction interval, same limitation as `forecast_gradient_boosted_trees` in `ts-deploy`.
- `forecaster/model_tools.py`'s `rolling_origin_backtest` reuses `fit_ets`/`fit_sarima`/`fit_gradient_boosted_trees` directly rather than reimplementing model fitting: it calls the SAME `fit_*` function repeatedly on progressively larger PREFIXES of `df` (`df.iloc[:test_end]`), relying on the fact that each `fit_*` function already does `train_test_split(df, holdout_size)` internally (last `holdout_size` rows = test, everything before = train) -- so truncating the input df to end at each origin automatically gives an expanding-window origin scheme for free, no separate windowing logic needed. `model_type` dispatch (`"ets"`/`"sarima"`/`"gbt"` -> the matching `fit_*` function, via `_ROLLING_ORIGIN_FIT_FUNCTIONS`) deliberately mirrors `retrain/retrain_tools.py::execute_redeploy`'s existing `model_type`/`params` dispatch convention for consistency across the codebase -- if you add a new `model_type` string to one, consider whether the other should know about it too. `"naive"` is deliberately NOT supported here (see the function's own docstring for why) and `fit_naive_baselines`' different two-baseline return shape isn't compatible with this function's single-result-per-origin assumption anyway.
- `forecaster/model_tools.py`'s `search_sarima_orders` is explicitly scoped as advisory, not authoritative — this was a deliberate design tension flagged before implementation: this toolkit's whole design (see `ts-forecaster/SKILL.md`) asks the agent to reason about model settings from `ts-analyst`'s findings, not grid-search blindly. Don't strengthen this tool's own claims (e.g. auto-selecting and returning a single "best" order rather than a `top_candidates` shortlist) without discussing that tension explicitly — it would work against the project's existing design principle, not just add a feature. `d`/`seasonal_d` are intentionally NOT searched (held fixed, caller-supplied) for the same reason. Verified directly that the numerically-best-AICc candidate on the project's own synthetic data has `residuals_look_like_white_noise: false` — a live, concrete example of why "ranked first by AICc" isn't the same as "actually good," baked into the tool's own docstring and `interpretation` field, not just asserted.
